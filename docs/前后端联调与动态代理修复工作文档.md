- 打通前端页面与后端真实接口，确保首页、登录注册、秒杀详情页、订单中心都不再依赖假数据。

## 3. Nginx 代理问题与修复

### 3.1 问题现象

宿主机通过 `http://127.0.0.1/api/orders/...` 访问订单服务时，出现间歇性：

- `502 Bad Gateway`
- `connect() failed (111: Connection refused) while connecting to upstream`

Nginx 日志中的上游 IP 为旧容器地址，例如：

- `upstream: "http://172.18.0.12:8000/api/orders/seckill"`

而订单服务当前实际 IP 已经变化为新地址。

### 3.2 根因分析

Docker Compose 在重建 `order_service`、`inventory_service` 等容器后，容器 IP 会变化。

Nginx 原配置虽然使用了服务名：

- `seckill_order_service`
- `seckill_inventory_service`

但在默认写法下，Nginx 会在启动时把服务名解析成 IP，之后继续使用旧 IP，不会自动跟踪 Docker DNS 的新解析结果。

因此出现：

- 容器服务本身是正常的
- 在容器内手工访问服务名可以成功
- 宿主机经由 Nginx 访问却继续命中旧 IP，从而返回 `502`

### 3.3 修复方案

修改 `nginx.conf`：

- 增加 Docker 内置 DNS：
  - `resolver 127.0.0.11 ipv6=off valid=30s;`
- 增加：
  - `resolver_timeout 5s;`
- 为所有上游服务节点加上 `resolve`
  - `server seckill_order_service:8000 resolve;`
  - `server seckill_inventory_service:8000 resolve;`
  - `server seckill_user:8000 resolve;`
  - `server seckill_product_1:8000 resolve;`
  - `server seckill_product_2:8000 resolve;`

这样 Nginx 会周期性重新解析 Docker 服务名，避免容器重建后仍使用旧 IP。

## 4. 环境联调说明

### 4.1 统一同源访问

前端生产环境配置改为：

- `frontend/.env.production`
  - `VITE_API_BASE_URL=`

这样浏览器中的所有请求都通过同源 `/api/...` 发往 Nginx，再由 Nginx 转发到各微服务。

优点：

- 避免前端直连多个端口
- 避免跨域配置分散
- 便于以后继续扩展网关策略

### 4.2 容器说明

`mysql-replica-init` 为一次性初始化容器。

它的正确行为是：

- 启动
- 等待主库和从库 ready
- 执行复制配置
- 以 `Exited (0)` 结束

因此它显示退出不代表失败，只要退出码为 0 就表示初始化完成。

本轮改造完成后，前端已不再停留在静态展示，而是接入了真实后端接口。

同时，Nginx 代理层的 Docker DNS 动态解析问题已被修复，能够避免因容器重建导致的旧 IP `502` 故障，前后端联调稳定性显著提升。
