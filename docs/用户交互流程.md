# 用户交互流程

**1. 用户购买一个商品的完整流程**
这是用户能看到的流程。

1. 用户进入首页 /，浏览商品、搜索商品、点进商品详情页。
2. 如果用户还没登录，点“立即购买”会被带去 /auth?redirect=... 登录页。
3. 登录成功后，前端把 JWT 保存到本地，之后再进入商品详情/秒杀页。
4. 商品详情页会同时加载两类信息：
   - 商品详情
   - 实时库存
5. 用户点击“立即购买”后，前端发起秒杀下单请求，并开始轮询订单状态。
6. 在用户视角里，订单会经历这些状态：
   - PENDING_INVENTORY：库存确认中
   - CREATED：订单已创建，可支付
   - PAYING：支付处理中
   - PAID：已支付
   - FAILED：失败
7. 如果订单变成 CREATED，用户就可以点“支付当前订单”。
8. 支付后前端继续轮询，直到订单变成 PAID。
9. 用户也可以去 /orders 订单中心查看全部订单、待处理订单、已支付订单，并继续支付未完成订单。

**2. 前端到后端，一个订单真实代码链路怎么走**
这是实际代码执行流程，不是页面文案流程。

1. 浏览器调用前端 axios 实例；http.js 会自动从 localStorage 取 access_token，加到 Authorization: Bearer ...。

2. 浏览器请求先到 Nginx：

   - /api/users 转给 user_service
   - /api/products 转给 product_service
   - /api/orders 转给 order_service
   - /api/inventory 转给 inventory_service

3. 登录链路：

   - 前端 AuthPortal.jsx 调 POST /api/users/login
   - user_service/main.py 校验用户名密码
   - 登录成功后生成 JWT，返回前端
   - 前端保存 JWT，后续请求自动带 Token

4. 商品详情页加载链路：

   - 前端

      

     FlashSaleArena.jsx

      

     并行请求：

     - GET /api/products/{id}
     - GET /api/inventory/products/{id}

   - 商品详情来自 product_service

   - 库存详情来自 inventory_service

5. 下单链路：

   - 前端调用 POST /api/orders/seckill

   - order_service

      

     先做三件事：

     - 解析 JWT，拿到当前用户
     - 检查“同一用户同一商品是否已有活动订单”
     - 去商品库确认商品存在

6. 然后 order_service 会同步调用库存服务内部接口：

   - POST /internal/inventory/reservations
   - 这一步是“先预扣库存，再落订单”

7. inventory_service 预扣库存时做两层控制：

   - Redis Lua 脚本原子执行

     - seckill:stock:{product_id}：库存数
     - seckill:user:{user_id}:product:{product_id}：用户限购标记
     - seckill:reservation:{order_id}：预留状态

   - MySQL

      

     inventory_db

     - 扣 inventory_item.available_stock
     - 加 inventory_item.reserved_stock
     - 写一条 inventory_reservation

8. 库存预扣成功后，order_service 才会真正落订单：

   - 写 t_order，状态是 PENDING_INVENTORY
   - 同时写 t_order_outbox_event，事件类型是 ORDER_CREATED

9. order_worker 异步发布 Outbox：

   - 把 ORDER_CREATED 发布到 Kafka topic：
   - seckill-order-created-topic

1. inventory_worker 消费这个 topic：

   - 把预留状态从 RESERVED 改成 CONFIRMED
   - 把 reserved_stock 转成 sold_stock
   - 再写一条库存侧 outbox 事件

2. inventory_worker 再把库存结果发到 Kafka：

   - topic：seckill-inventory-result-topic

3. order_worker 消费库存结果：

   - 如果成功，把订单状态改成 CREATED
   - 并写入 t_user_purchase_record
   - 这张表用来保证“同一用户同一商品只能成功购买一次”

4. 前端轮询 GET /api/orders/{order_id}，看到状态变成 CREATED，就展示支付按钮。

5. 支付链路：

   - 前端调用 POST /api/orders/{order_id}/pay

   - order_service

      

     校验：

     - 订单存在
     - 订单不是失败状态
     - 库存已确认
     - 支付金额等于订单金额

   - 然后写：

     - t_payment，状态 PROCESSING
     - t_order_outbox_event，事件类型 PAYMENT_REQUESTED
     - 同时把订单状态改成 PAYING

6. order_worker 把支付事件发到 Kafka：

   - topic：seckill-payment-topic

7. 目前没有独立支付服务。

   - 现在是 order_worker 自己又消费这个 payment topic
   - 然后把：
     - t_payment.status 改成 SUCCESS
     - t_order.status 改成 PAID
   - 所以当前支付是“异步模拟成功”的实现，不是第三方支付网关

8. 前端继续轮询，最后看到 PAID，流程结束。

**3. 目前所有服务的职责**
我分成业务服务和基础设施服务两类说。

**业务服务**

- frontend_nginx
  - 承载前端静态页面
  - 统一把 /api/* 请求反向代理到后端各服务
- user_service
  - 用户注册
  - 用户登录
  - bcrypt 密码加密
  - JWT 签发
- product_service_1
  - 商品列表、商品详情、商品搜索
  - Redis 缓存
  - 读写分离中的读实例服务
- product_service_2
  - 与 product_service_1 同职责
  - 主要为了商品读请求负载均衡
- order_service
  - 对外提供下单、支付、查单、订单列表接口
  - 订单写库
  - 调库存服务预扣库存
- order_worker
  - 发布订单 outbox 到 Kafka
  - 消费库存确认结果
  - 消费支付结果
  - 异步更新订单状态
- inventory_service
  - 对外提供库存查询
  - 对内提供库存预留与取消接口
  - Redis 原子预扣减
  - 库存 MySQL 落库
- inventory_worker
  - 消费订单创建事件
  - 确认库存预留
  - 发布库存结果事件

**基础设施服务**

- mysql-primary
  - 主库
  - 承担所有写入
  - 也承担一部分读取
- mysql-replica
  - 从库
  - 主要给商品服务读请求做读扩展
- mysql-replica-init
  - 一次性初始化容器
  - 只负责把主从复制配起来
- shardingsphere-proxy
  - 给订单库提供逻辑库 order_db
  - 屏蔽底层 order_db_0 / order_db_1 分片细节
- redis
  - 商品缓存
  - 秒杀库存缓存
  - 用户限购状态
  - 订单/预留临时状态
- kafka
  - 异步消息总线
  - 连接订单服务和库存服务
- zookeeper
  - Kafka 依赖组件
- api_gateway/
  - 代码目录里有预留入口
  - 当前实际流量并不是它承接，真正网关层是 Nginx

**4. 目前各个数据库的职能，以及表里记录什么**
这里建议把“关系型数据库”和“Redis”分开理解。

**4.1 seckill_db**
这是商品库，也是用户服务当前使用的库。
物理上在 mysql-primary，并复制到 mysql-replica。

表：

- seckill_product
  - 商品基础信息
  - 字段核心含义：
    - id：商品 ID
    - name：商品名
    - price：价格
    - stock：基础库存字段
- users
  - 用户账号信息
  - 字段核心含义：
    - id：用户 ID
    - username：用户名
    - password_hash：加密后的密码
    - created_at：注册时间

说明：

- users 表不是初始化 SQL 里建的，而是 user_service 启动时自动建表。
- product_service 读这个库时优先走副本库，失败再回退主库。

**4.2 inventory_db**
这是库存服务自己的数据库。

表：

- inventory_item
  - 每个商品的库存主账本
  - 记录：
    - product_id
    - product_name
    - unit_price
    - available_stock：可售库存
    - reserved_stock：已预留未最终成交
    - sold_stock：已售
    - version：并发控制版本号
- inventory_reservation
  - 每个订单对应的一条库存预留记录
  - 记录：
    - order_id
    - user_id
    - product_id
    - quantity
    - status：RESERVED / CONFIRMED / CANCELED
    - failure_reason
- inventory_outbox_event
  - 库存服务待发布/已发布的消息事件表
  - 记录：
    - 事件属于哪个订单 aggregate_id
    - 发往哪个 Kafka topic
    - 事件类型
    - payload
    - 发布状态、重试次数、下次重试时间

**4.3 逻辑库 order_db**
应用看到的是 order_db，实际底层是分片后的：

- order_db_0
- order_db_1

ShardingSphere 规则：

- 按 user_id % 2 决定落到 ds_0 还是 ds_1
- t_order / t_payment / t_order_outbox_event 再按 order_id 或 aggregate_id 分到 _0 / _1 表

所以代码里写的是：

- t_order
- t_payment
- t_order_outbox_event
- t_user_purchase_record

实际物理表是：

- t_order_0 / t_order_1
- t_payment_0 / t_payment_1
- t_order_outbox_event_0 / t_order_outbox_event_1
- t_user_purchase_record

表职责：

- t_order
  - 订单主表
  - 记录：
    - order_id
    - user_id
    - product_id
    - quantity
    - unit_price
    - total_amount
    - status
    - failure_reason
    - created_at / updated_at
- t_payment
  - 支付记录表
  - 记录：
    - order_id
    - user_id
    - amount
    - status
    - 时间戳
- t_order_outbox_event
  - 订单服务的 outbox 事件表
  - 记录：
    - 订单创建事件
    - 支付请求事件
    - 发布状态、重试次数、发布时间
- t_user_purchase_record
  - 用户购买去重表
  - 记录：
    - (user_id, product_id) 唯一组合
    - 对应 order_id
  - 作用是防止用户对同一商品重复成功下单

**4.4 Redis**
严格说它不是关系型数据库，但在这套系统里它承担了非常重要的“实时状态数据库”角色。

商品缓存相关 key：

- product:list:page:{page}:size:{size}:keyword:{keyword}
- product:detail:{product_id}

秒杀库存相关 key：

- seckill:stock:{product_id}
  - 当前可抢库存
- seckill:user:{user_id}:product:{product_id}
  - 用户限购标记
- seckill:reservation:{order_id}
  - 某个订单的库存预留状态

作用：

- 商品服务缓存热点商品
- 秒杀库存用 Lua 脚本做原子扣减
- 防超卖
- 防同一用户重复抢购
- 保存短期状态，减轻 MySQL 压力

**5. 你现在这套系统里，一个订单最重要的几个状态转移**
你汇报时可以直接这样说：

- 下单请求进入后，先预扣库存，再写订单
- 订单先进入 PENDING_INVENTORY
- 库存服务异步确认后，订单进入 CREATED
- 用户支付后，订单进入 PAYING
- 支付异步确认后，订单进入 PAID
- 任一关键环节失败，订单进入 FAILED

**6. 当前实现里一个很重要的“真实现状”**
这个点建议你对老师或团队成员说明清楚：

- 现在已经有完整的异步订单链路、库存链路、分库分表、Redis 原子预扣减
- 但“支付”目前不是第三方支付网关
- 它是由 order_service 自己发消息、自己消费消息、异步把支付状态改成成功
- 所以这是“支付异步流程模拟完成版”，不是“真实支付渠道接入版”
