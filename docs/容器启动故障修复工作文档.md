# 容器启动故障修复工作文档

## 1. 问题概览

本次 `docker compose up --build` 失败并非单点故障，而是由以下几个独立问题叠加导致：

1. `shardingsphere-proxy` 配置文件不兼容当前镜像版本，启动时解析 YAML 失败。
2. `zookeeper` 实际已启动，但健康检查使用了未启用的四字命令，导致一直被判定为 `unhealthy`。
3. `mysql-replica` 在初始化阶段带着 `--read-only=ON` 与 `--super-read-only=ON` 启动，初始化 SQL 无法执行。
4. `kafka` 同时声明了两个监听器，但都绑定到了 `9092` 端口，启动时直接退出。

## 2. 修复项

### 2.1 ShardingSphere 配置修复

文件：`infra/shardingsphere/conf/config-sharding.yaml`

- 移除了顶层不被 `apache/shardingsphere-proxy:5.3.2` 支持的 `props` 配置块。
- 修复后 `shardingsphere-proxy` 可正常启动。

### 2.2 Zookeeper 健康检查修复

文件：`docker-compose.yml`

- 将健康检查从 `ruok` 改为当前镜像中已启用的 `srvr` 指令。
- 保留并放宽了启动缓冲时间：
  - `start_period: 30s`
  - `retries: 12`

修复后 `zookeeper` 状态恢复为 `healthy`。

### 2.3 MySQL 主从初始化修复

文件：`docker-compose.yml`

- 删除了 `mysql-replica` 启动参数中的：
  - `--read-only=ON`
  - `--super-read-only=ON`

文件：`infra/mysql/scripts/setup-replica.sh`

- 在执行复制配置前，先临时关闭：
  - `SET GLOBAL super_read_only = OFF;`
  - `SET GLOBAL read_only = OFF;`
- 复制配置完成后再重新开启只读：
  - `SET GLOBAL read_only = ON;`
  - `SET GLOBAL super_read_only = ON;`

修复后副本库可完成初始化并成功建立主从复制。

### 2.4 Kafka 监听器修复

文件：`docker-compose.yml`

- 为 Kafka 显式补充：
  - `KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,PLAINTEXT_HOST://0.0.0.0:29092`
- 保留原有对外声明：
  - `KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092,PLAINTEXT_HOST://localhost:9092`
- 将宿主机端口映射改为：
  - `9092:29092`

这样容器内微服务继续访问 `kafka:9092`，宿主机仍可通过 `localhost:9092` 访问 Kafka。
